# 12 进阶概念补充

## 1. 这一节要解决什么问题

这一节补充几个日常协作中很实用但前面没详细讲的概念：`git stash`、`.gitignore`、`git diff`、`git tag`、`fork` 与 `clone` 的区别，以及 `git merge` 和 `git rebase` 的基本区别。

## 2. 基础概念解释

### git stash（临时储藏）

通俗解释：stash 像"暂时放进抽屉"，你临时需要切换去干别的，但当前工作没做完不想 commit，就先把改动放进抽屉，回来再取出来继续。

技术解释：`git stash` 把工作区和暂存区的未提交改动存入一个临时栈，工作区恢复干净状态，之后 `git stash pop` 可以取回。

例子：你在 feat/A 分支上改到一半，老师突然说要在 feat/B 分支上修一个紧急问题，先 stash 再切分支。

### .gitignore（忽略文件）

通俗解释：`.gitignore` 是一份"黑名单"，告诉 Git 哪些文件永远不要追踪和提交，比如你电脑的私密配置、自动生成的缓存文件。

技术解释：`.gitignore` 是仓库根目录下的一个文本文件，里面的路径和模式匹配到的文件会被 Git 完全忽略，不显示在 `git status` 里，也不会被 commit。

例子：`.env`（包含数据库密码）、`node_modules/`（可以重新安装）、`__pycache__/`（Python 缓存）都应该写进 `.gitignore`。

### git tag（标签）

通俗解释：tag 像给某个时刻盖一个永久印章，比如"v1.0 正式发布"，以后想回到这个版本直接找这个印章就行。

技术解释：tag 是指向某个特定 commit 的不可移动引用（与分支不同，分支随新 commit 移动，tag 固定不动），通常用于版本发布标记。

例子：`git tag v1.0.0` 在当前 commit 上打一个 v1.0.0 的标签。

### fork（派生）

通俗解释：fork 是"在 GitHub 上复制别人的仓库到自己的账号下"，相当于把公共图书馆的一本书复印一份，放进你自己的书柜，随便改都行，不影响图书馆原本。

技术解释：fork 是 GitHub 的操作，在你的账号下创建一个目标仓库的完整副本，你对这个副本有完全权限，可以通过 PR 向原仓库贡献代码。

例子：你想给一个开源项目贡献代码：fork → clone 你自己的 fork → 修改 → PR → 原仓库 maintainer 审查合并。

## 3. 为什么要学这个

这些概念虽然不是每天都用，但遇到时如果不知道，就会很困惑：为什么切分支报错要先 stash？为什么有些文件不在 git status 里？tag 和 branch 有什么区别？这节补齐这些疑惑。

## 4. 关键知识点

### 4.1 git stash 用法

解释：stash 是临时保存的好工具，比 `git reset --hard` 安全得多。

命令：

```powershell
# 保存当前所有未提交改动（工作区 + 暂存区）
git stash

# 保存时附加说明
git stash push -m "feat/A 写到一半，临时切去修紧急 bug"

# 查看 stash 列表
git stash list
# 输出：
# stash@{0}: On feat/A: feat/A 写到一半，临时切去修紧急 bug
# stash@{1}: WIP on main: 3a7b1d9 docs: 上次的改动

# 恢复最近一次 stash（并从列表删除）
git stash pop

# 恢复但不删除（保留在列表里）
git stash apply stash@{0}

# 删除某个 stash
git stash drop stash@{0}

# 清空所有 stash
git stash clear
```

注意事项：`stash pop` 时如果恢复的改动和当前工作区有冲突，会出现 CONFLICT，按照 10 节的方法处理。

### 4.2 .gitignore 用法

解释：`.gitignore` 要在第一次 commit 之前就加好，已经追踪的文件加进去没有效果。

常见内容模板：

```gitignore
# Python
__pycache__/
*.pyc
.venv/
.env

# Node.js
node_modules/
dist/
.env.local

# 编辑器
.vscode/
.idea/
*.swp

# 系统文件
.DS_Store
Thumbs.db

# 日志
*.log
logs/
```

命令：

```powershell
# 检查某个文件是否被 ignore
git check-ignore -v .env

# 如果文件已经被追踪，需要先移除追踪再 ignore
git rm --cached .env
# 然后把 .env 加进 .gitignore
```

注意事项：`.env` 文件里通常有数据库密码、API Key，一旦提交到 GitHub（尤其是公开仓库），密钥就泄露了。这是新手最常见的安全问题之一。

### 4.3 git diff 用法

解释：diff 是"比较差异"，可以比较不同区域或不同分支之间的变化。

命令：

```powershell
# 工作区 vs 暂存区（改了但还没 add 的内容）
git diff

# 暂存区 vs 最近一次 commit（已 add 但还没 commit 的内容）
git diff --staged

# 比较两个分支的差异
git diff main feat/weekly-tanglx-w26

# 比较某个文件的差异
git diff docs/weekly/2026-W26.md

# 比较两个 commit
git diff abc1234 def5678
```

输出结果怎么看：

```text
diff --git a/docs/weekly/2026-W26.md b/docs/weekly/2026-W26.md
index 3a7b1d9..f2e8c1a 100644
--- a/docs/weekly/2026-W26.md    ← 旧版本
+++ b/docs/weekly/2026-W26.md    ← 新版本
@@ -1,3 +1,5 @@
 # 2026-W26 周报
+
+## 本周完成
+- 学习了 Git 基础操作    ← + 号是新增的行
-旧内容                          ← - 号是删除的行
```

### 4.4 git tag 用法

解释：tag 通常用于标记版本发布节点，和分支不同，tag 不会随新 commit 移动。

命令：

```powershell
# 在当前 commit 打轻量 tag
git tag v1.0.0

# 打注释 tag（推荐，包含说明信息）
git tag -a v1.0.0 -m "第一个正式版本"

# 查看所有 tag
git tag

# 查看某个 tag 的详细信息
git show v1.0.0

# 推送 tag 到 GitHub
git push origin v1.0.0

# 推送所有 tag
git push origin --tags

# 删除本地 tag
git tag -d v1.0.0

# 删除远端 tag
git push origin :refs/tags/v1.0.0
```

注意事项：tag 和分支的区别：分支随新 commit 自动前移，tag 固定在某个 commit 上不动。

### 4.5 fork 和 clone 的区别

解释：两者都是"复制仓库"，但操作在不同地方，用途也不同。

例子：

| 操作 | 在哪里发生 | 结果 | 适用场景 |
| --- | --- | --- | --- |
| clone | 本地执行，把远端复制到本机 | 你电脑上有一份仓库副本 | 团队协作、日常工作 |
| fork | GitHub 网页上操作 | 你的 GitHub 账号下有一份仓库副本 | 向开源项目贡献代码 |

开源贡献标准流程：

```text
1. GitHub 上 fork 原始仓库到你的账号
2. clone 你自己的 fork 到本地
3. 新建功能分支，修改代码
4. push 到你的 fork
5. 在 GitHub 上向原仓库发起 PR
6. 原仓库 maintainer review 后 merge
```

注意事项：fork 出来的仓库和原仓库是独立的，你对 fork 仓库有完全权限，改坏了也不影响原仓库。

### 4.6 git merge 和 git rebase 的基本区别

解释：两者都是"把分支内容合并进来"，但历史记录形态不同。

例子：

```text
情景：从 main 的 C 点派生出 feat 分支，做了 D、E，
      同时 main 又有了 F 提交

git merge：
  main: A - B - C - F - M（M 是合并提交）
                      ↗
  feat: D - E ───────

  特点：保留了分叉历史，有一个额外的"合并提交"

git rebase：
  把 D、E 移植到 F 之后：
  main: A - B - C - F - D' - E'
  
  特点：历史是一条直线，更干净，但改写了 D、E 的 commit hash
```

新手建议：

```text
- 日常用 git pull（内部是 merge）就足够了
- rebase 改写历史，用在共享分支上很危险
- 团队有规范就遵循团队规范
- 新手阶段先把 merge 理解透，rebase 以后再学
```

## 5. 和前后知识的关系

本节是对前面所有内容的补充和拓展。stash 是 pull/切换分支时的安全工具，.gitignore 是项目初始化时必做的配置，tag 是项目版本管理的工具，fork 是开源贡献的入口。

## 8. 常见误区

1. 误区：.gitignore 可以后加，任何时候加都有效。  
   解释：已经被 Git 追踪过的文件，加进 .gitignore 不会生效，需要先 `git rm --cached`。

2. 误区：stash 里的内容永久保存了，不用担心丢失。  
   解释：stash 存在本地，如果仓库被删除，stash 也会消失。重要改动应该 commit。

3. 误区：fork 和 clone 是一回事。  
   解释：fork 在 GitHub 上发生，是账号间的仓库复制；clone 在本地执行，是把远端仓库下载到本机。

4. 误区：rebase 比 merge 更好，应该总用 rebase。  
   解释：rebase 改写历史，在共享分支上用是危险操作。团队要统一规范，新手先用 merge。

## 9. 本节总结

`git stash` 是临时保存工作的安全方式，`git stash pop` 恢复；`.gitignore` 要提前配置，避免提交密码和缓存文件；`git diff` 查看各区域差异；tag 是版本快照，固定不动；fork 是 GitHub 账号间的仓库复制，用于开源贡献；rebase 新手暂时不用学，merge 理解清楚就够了。
