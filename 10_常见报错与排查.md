# 10 常见报错与排查

## 1. 这一节要解决什么问题

这一节收录最常见的 Git 报错信息，每条都给出原因解释和解决步骤，让你遇到报错时不慌，能自己判断和处理。

## 2. 基础概念解释

### 报错的基本判断原则

通俗解释：Git 报错信息通常很直白，关键词就说明了问题。先看关键词，再查原因，再对症下药。

技术解释：Git 错误通常分三类：认证失败（SSH/权限问题）、状态冲突（本地和远端不一致）、操作错误（命令用错了）。

例子：看到 `Permission denied` 就是权限/SSH 问题；看到 `rejected` 就是推送被拒绝；看到 `CONFLICT` 就是合并冲突。

## 3. 为什么要学这个

新手遇到报错最容易的反应是"复制命令再执行一遍"或者随机尝试，这会让问题越来越复杂。理解报错原因，才能对症处理，不会越弄越乱。

## 4. 关键知识点

### 4.1 Permission denied (publickey)

报错信息：

```text
git@github.com: Permission denied (publickey).
fatal: Could not read from remote repository.
```

原因：SSH Key 没有配置，或公钥没有添加到 GitHub，或用了 HTTPS 地址但 SSH 没有配置。

解决步骤：

```powershell
# 第一步：测试 SSH 连接
ssh -T git@github.com

# 如果输出 "Permission denied"，说明 SSH Key 没配好

# 第二步：确认 SSH Key 存在
ls ~/.ssh/

# 如果没有 id_ed25519 或 id_rsa，需要生成 Key（参考 02 节）

# 第三步：确认公钥已添加到 GitHub
# 打开 GitHub Settings → SSH and GPG keys，查看是否有你的 Key

# 第四步：确认 remote 地址用的是 SSH 格式
git remote -v
# 应该是 git@github.com:xxx/xxx.git
# 如果是 https:// 开头，切换为 SSH：
git remote set-url origin git@github.com:Firebat/zhongwen-shortdrama-context.git
```

注意事项：最常见原因是公钥没有正确添加到 GitHub，或者 Key 名不是默认的 id_rsa/id_ed25519。

### 4.2 fatal: not a git repository

报错信息：

```text
fatal: not a git repository (or any of the parent directories): .git
```

原因：你当前所在目录不是 Git 仓库（没有 `.git` 目录）。

解决步骤：

```powershell
# 确认当前目录
pwd

# 进入正确的项目目录
cd C:\Users\Firebat\zhongwen-shortdrama-context

# 再试
git status
```

注意事项：如果确实想在当前目录初始化一个新仓库，用 `git init`，但对于已有的团队项目，应该 clone 而不是 init。

### 4.3 error: failed to push some refs / rejected (non-fast-forward)

报错信息：

```text
 ! [rejected]        main -> main (non-fast-forward)
error: failed to push some refs to 'github.com:Firebat/xxx.git'
hint: Updates were rejected because the remote contains work that you do not have locally.
```

原因：远端有你本地没有的提交（队友 push 了新代码），直接 push 会覆盖别人的工作，所以被拒绝。

解决步骤：

```powershell
# 第一步：先 pull，把远端的新内容合并到本地
git pull

# 如果 pull 成功（没有冲突），再 push
git push

# 如果 pull 出现冲突，参考 4.4 节处理冲突
```

注意事项：⚠️ 不要用 `git push -f` 来"解决"这个问题，这会覆盖别人的提交，是破坏性操作。

### 4.4 CONFLICT (content): Merge conflict

报错信息：

```text
Auto-merging docs/weekly/2026-W26.md
CONFLICT (content): Merge conflict in docs/weekly/2026-W26.md
Automatic merge failed; fix conflicts and then commit the result.
```

原因：你和队友修改了同一个文件的同一个位置，Git 无法自动决定用哪个版本，需要你手动选择。

冲突文件内容示例：

```text
<<<<<<< HEAD
这是你本地的内容
=======
这是队友/远端的内容
>>>>>>> origin/main
```

解决步骤：

```powershell
# 第一步：查看有哪些冲突文件
git status
# 显示 "both modified: docs/weekly/2026-W26.md"

# 第二步：打开冲突文件，手动编辑
# 删掉 <<<<<, =====, >>>>> 这三行，保留你想要的内容

# 第三步：保存文件后，标记冲突已解决
git add docs/weekly/2026-W26.md

# 第四步：提交合并结果
git commit -m "merge: 解决 2026-W26 周报冲突"

# 然后可以 push
git push
```

注意事项：解决冲突前一定要理解两边改了什么，不要随便删掉队友的内容。如果不确定，和队友沟通再处理。

### 4.5 Your local changes would be overwritten by merge

报错信息：

```text
error: Your local changes to the following files would be overwritten by merge:
        docs/weekly/2026-W26.md
Please commit your changes or stash them before you merge.
Aborting
```

原因：工作区有未提交的修改，pull 会覆盖它，所以 Git 拒绝操作，保护你的修改。

解决步骤：

```powershell
# 方案1：先暂存改动，pull 后恢复（推荐）
git stash
git pull
git stash pop

# 方案2：先 commit，再 pull
git add .
git commit -m "wip: 临时保存进度"
git pull
```

注意事项：`git stash` 是临时保存的好工具，`stash pop` 恢复后如果有冲突，和 4.4 节同样处理。

### 4.6 detached HEAD state

报错信息：

```text
HEAD detached at 3a7b1d9
```

原因：你 `git checkout` 了一个具体的 commit hash，而不是分支名，HEAD 不再指向任何分支。

解决步骤：

```powershell
# 查看当前状态
git status

# 方案1：如果你只是想查看，直接切回分支
git switch main
# 或
git switch -

# 方案2：如果你在 detached HEAD 状态做了 commit，想保留
git switch -c new-branch-name  # 创建新分支保存这些 commit
```

注意事项：detached HEAD 状态下做的 commit，如果直接切换分支，这些 commit 会"丢失"（不会立刻消失，但没有分支引用会指向它们）。

### 4.7 refusing to merge unrelated histories

报错信息：

```text
fatal: refusing to merge unrelated histories
```

原因：两个仓库没有共同的祖先 commit，通常出现在：本地用 `git init` 初始化了一个仓库，又想和 GitHub 上另一个已有历史的仓库关联。

解决步骤：

```powershell
# 允许合并不相关历史（用于一次性的初始化场景）
git pull origin main --allow-unrelated-histories

# 注意：这个命令要谨慎，确认两边的代码你都想保留
```

注意事项：正常流程（clone → 工作）不会遇到这个问题。只有错误地用了 `git init` + `git remote add` 流程时才会出现。应该用 `git clone` 代替。

## 8. 常见误区

1. 误区：报错了就用 `git reset --hard` 清空。  
   解释：`git reset --hard` 会丢弃所有未提交改动，是破坏性操作。报错应该先理解原因再处理。

2. 误区：冲突了就随便删掉一边。  
   解释：冲突意味着两边都有修改，随便删掉一边可能丢失队友的重要改动。要理解两边改了什么。

3. 误区：`git push -f` 能解决所有 push 失败。  
   解释：`git push -f` 会覆盖远端历史，在共享分支上会破坏队友的工作。正确做法是先 pull。

## 9. 本节总结

遇到 Git 报错的第一反应：先看报错关键词，判断是权限问题、同步问题还是冲突。Permission denied 查 SSH 配置；rejected 先 pull；CONFLICT 手动解决后 commit；local changes would be overwritten 先 stash 或 commit。不要随便用 `git push -f` 或 `git reset --hard`。
