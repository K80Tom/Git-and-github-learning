# 11 危险命令与安全习惯

## 1. 这一节要解决什么问题

这一节列出所有新手容易误用的危险命令，解释每个命令为什么危险、会带来什么后果，以及什么时候才可以用、不能用时的安全替代方案。

## 2. 基础概念解释

### 危险命令的定义

通俗解释：危险命令是那些"执行后无法撤销"或"会影响到别人代码"的命令，就像生活里的"不可逆操作"——电话挂了、文件删了、蛋打了。

技术解释：Git 中的危险操作主要是两类：丢弃本地数据（reset --hard、clean -fd）和改写远端历史（push -f）。这两类操作要么无法恢复本地修改，要么破坏团队共享的提交历史。

## 3. 为什么要学这个

新手出问题大多数是因为不知道某个命令有破坏性，随便执行了。提前知道哪些命令不能碰，比出了问题再补救好得多。

## 4. 关键知识点

### 4.1 ⛔ git push -f / git push --force

危险级别：⛔⛔⛔ 最高危险

危险原因：强制把本地历史推送到远端，覆盖远端的提交历史。如果队友刚 push 了重要代码，你一个 `push -f`，他们的代码在远端就消失了。

```text
场景：
  远端：A - B - C - D（D 是队友刚 push 的）
  你本地：A - B - E  （你改写了历史）
  git push -f 之后：
  远端：A - B - E    ← D 消失了，队友的工作丢失
```

什么时候才能用：只有你独自使用的临时分支，且你明确知道要丢弃的内容是什么，才可以考虑。团队共享分支（main、develop 等）绝对不能用。

安全替代方案：

```powershell
# 不要用 push -f，先 pull 再 push
git pull
git push
```

> ⚠️ **新手不要随便用 `git push -f`**

---

### 4.2 ⛔ git reset --hard

危险级别：⛔⛔ 高危险

危险原因：把工作区、暂存区、当前分支都强制回退到指定状态，**丢弃所有未提交的修改，且无法找回**。

```powershell
# 危险：丢弃所有未提交修改（工作区 + 暂存区全部清空）
git reset --hard HEAD

# 危险：回退到某个旧 commit，新的 commit 也不见了
git reset --hard abc1234
```

什么时候才能用：你确认要彻底放弃当前所有未提交修改，没有任何需要保留的改动时。

安全替代方案：

```powershell
# 想临时保存改动 → git stash（可以恢复）
git stash
git stash pop  # 恢复

# 只想丢弃单个文件的修改（比 reset --hard 更精确）
git restore 文件名
```

> ⚠️ **新手不要随便用 `git reset --hard`，先用 `git stash` 代替**

---

### 4.3 ⚠️ git checkout -- `<file>` / git restore `<file>`

危险级别：⚠️ 中等危险

危险原因：丢弃单个文件在工作区的所有未提交修改，**无法找回**。

```powershell
# 丢弃 文件 在工作区的修改（旧语法）
git checkout -- docs/weekly/2026-W26.md

# 新语法（效果相同）
git restore docs/weekly/2026-W26.md
```

什么时候才能用：你确认这个文件的修改完全不需要，想恢复到最近一次 commit 的状态时。

安全做法：操作前先用 `git diff 文件名` 看清楚改了什么，确认不需要再执行。

```powershell
# 先看改了什么
git diff docs/weekly/2026-W26.md

# 确认不需要后再丢弃
git restore docs/weekly/2026-W26.md
```

---

### 4.4 ⛔ git clean -fd

危险级别：⛔⛔ 高危险

危险原因：删除所有未被 Git 追踪的文件和目录（Untracked files），**无法恢复**。

```powershell
# 危险：删除所有未追踪文件
git clean -fd
```

什么时候才能用：你确认要清除所有"新建但未 add 过"的文件，比如清理构建生成的临时文件时。

安全替代方案：

```powershell
# 先看会删哪些文件（-n 是演练模式，不真正删除）
git clean -fdn

# 确认没问题后再真正执行
git clean -fd
```

> ⚠️ **执行 `git clean -fd` 之前，一定先运行 `git clean -fdn` 预览**

---

### 4.5 ⛔ 直接在 main 上修改并 push

危险级别：⛔⛔⛔ 最高危险（团队场景）

危险原因：
- 绕过了 PR 和 code review，没有人审查你的改动
- 如果改动有 bug，直接影响所有人的 main
- 很多团队仓库有保护规则，直接 push main 会被拒绝

```text
❌ 错误流程：
  git switch main → 修改文件 → git commit → git push main

✅ 正确流程：
  git switch main → git pull → git switch -c feat/xxx → 修改 → commit → push → PR
```

---

### 4.6 ⚠️ git merge / git rebase（在不理解时）

危险级别：⚠️ 中等危险（新手不懂时）

危险原因：
- `git merge` 不当可能产生复杂的合并提交
- `git rebase` 会改写 commit 历史，用在共享分支上和 `push -f` 一样危险

新手建议：

```text
- git merge：拉取代码用 git pull（内部是 fetch + merge），不要手动 merge 不熟悉的分支
- git rebase：暂时不要用，等理解了 commit 历史之后再学
```

---

### 4.7 安全习惯总结

解释：建立这些习惯，能避免 80% 的 Git 灾难。

例子：

```text
✅ 操作前先 git status，确认当前状态
✅ 不确定的命令先查文档，不要乱试
✅ 重要改动先 git stash 或 git commit 保存，再做危险操作
✅ 不在 main 上直接修改，永远用 feature branch
✅ push 之前先 pull，避免覆盖队友的工作
✅ git clean -fd 之前先用 -n 预览
✅ 不懂的情况下，绝对不用 git push -f
✅ 不在共享分支（main）上做 rebase
```

## 5. 和前后知识的关系

本节是安全守则，贯穿整个学习过程。每次遇到报错想"用危险命令来解决"时，先来查这一节确认是否合适。

## 6. 实战任务

把以下危险命令列表打印出来或保存，贴在你的工作桌面：

```text
⛔ 团队协作中绝对不要用：
  - git push -f（共享分支上）
  - 直接 push main
  
⚠️ 操作前必须确认的：
  - git reset --hard（先考虑 git stash）
  - git clean -fd（先用 -n 预览）
  - git checkout -- 文件名（先 git diff 确认）
```

3. `git stash` 相比 `git reset --hard` 的优势是什么？
4. `git clean -fd` 删除的是哪类文件？
5. 如果想清理未追踪文件，操作前应该先运行什么命令？

## 8. 常见误区

1. 误区：`git reset --hard` 可以"撤销" push 过的 commit。  
   解释：`reset --hard` 只影响本地，已经 push 到远端的 commit 需要用 `git push -f` 才能覆盖（这更危险）。

2. 误区：我只是改改本地，不影响别人，可以随便 reset。  
   解释：`git reset --hard` 本地的改动丢了就真的丢了，无法找回。

3. 误区：`git clean` 只删空文件，不影响重要内容。  
   解释：`git clean -fd` 删除所有未追踪的文件和目录，你新建的但还没 add 的文件也会被删掉。

## 9. 本节总结

危险命令的核心原则：凡是"不可逆"或"影响别人历史"的命令，都需要在充分理解的情况下才能使用。新手阶段的安全守则：不用 `git push -f`，不在 main 上直接推送，`git reset --hard` 先考虑 `git stash` 替代，`git clean` 先用 `-n` 预览。
