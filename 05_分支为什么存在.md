# 05 分支为什么存在

## 1. 这一节要解决什么问题

这一节让你理解分支（branch）为什么存在、main 分支为什么要保护、feature branch 是什么，以及如何创建和切换分支。理解了分支，才能理解为什么团队协作不能直接改 main。

## 2. 基础概念解释

### Branch（分支）

通俗解释：分支像平行宇宙。在主线（main）上开一个分支，就是开了一条平行时间线，你在这条线上的改动不影响主线，做完了再合并回去。

技术解释：分支是指向某个 commit 的可移动指针。创建分支不复制文件，只是创建一个新指针，非常轻量。

例子：你新建 `feat/weekly-tanglx-report` 分支，在这个分支上改文件，main 分支完全不受影响。

### main 分支

通俗解释：main 是项目的"主线剧情"，代表目前最稳定、最正式的版本。

技术解释：main（旧称 master）是 Git 仓库的默认主分支，通常代表生产可用的稳定代码，团队约定不直接在 main 上做未经审查的修改。

例子：老师、同学、所有人看到的"正式版本"都是 main，你的个人改动都应该在自己的 feature branch 上。

### Feature Branch（功能分支）

通俗解释：功能分支是你为了完成一个具体任务而新建的专属分支，用完就可以删掉。

技术解释：Feature branch 是从 main 或其他分支派生出来的临时工作分支，命名通常包含任务类型和描述，如 `feat/xxx`、`fix/xxx`、`docs/xxx`。

例子：写周报 → 建 `docs/weekly-tanglx-w26`；修一个 bug → 建 `fix/login-error`。

### HEAD

通俗解释：HEAD 是"你现在在哪里"的标记，像书签一样插在当前分支的最新 commit 上。

技术解释：HEAD 是 Git 中指向当前所在 commit 或分支的引用，通常指向某个分支名，切换分支就是移动 HEAD。

例子：`git log` 里显示 `(HEAD -> feat/weekly-tanglx-report)` 说明你现在在这个分支上。

### Remote Branch（远端分支）

通俗解释：远端分支是 GitHub 上的分支，你 push 之后 GitHub 才有，不 push 的话只存在你本地。

技术解释：远端分支是存在远端仓库（origin）上的分支，在本地用 `origin/xxx` 形式表示追踪信息。

例子：你本地有 `feat/my-task`，push 之后 GitHub 上也有 `feat/my-task`，本地显示为 `origin/feat/my-task`。

## 3. 为什么要学这个

如果不理解分支，你就不明白为什么不能直接在 main 上改东西，也不明白 PR 的意义。分支让团队成员可以并行工作，互不干扰，改完了再汇总审查，是团队协作的核心机制。

## 4. 关键知识点

### 4.1 为什么不要直接在 main 上改东西

解释：直接在 main 上改东西，会绕过 review 流程，影响所有人，出错无法隔离。

例子：

```text
错误做法（直接在 main 上改）：
  main: A - B - C - [你的改动D]  ← 直接改了 main
  问题1：绕过了 PR，没有人 review 你的改动
  问题2：如果 D 有 bug，所有人的 main 都受影响
  问题3：很多团队仓库会保护 main，直接 push 会被拒绝

正确做法（建功能分支）：
  main: A - B - C              ← main 不受影响
  feat/xxx: A - B - C - D      ← 你的改动在这里
  改完 → push feat/xxx → 开 PR → review → merge 进 main
```

注意事项：团队项目的 main 通常设置了 Branch Protection Rules，直接 push main 会被 GitHub 拒绝。

### 4.2 分支操作常用命令

解释：现代 Git 推荐用 `git switch` 来创建和切换分支，比旧的 `git checkout` 更直观。

命令：

```powershell
# 查看所有本地分支（当前分支前有 * 标记）
git branch

# 查看所有分支（含远端）
git branch -a

# 切换到已有分支
git switch main

# 新建分支（从当前位置）
git switch -c feat/weekly-tanglx-report

# 新建分支（从指定分支）
git switch -c feat/my-task origin/main

# 删除本地分支（安全版，只有已合并才能删）
git branch -d feat/weekly-tanglx-report

# 强制删除本地分支（谨慎使用）
git branch -D feat/weekly-tanglx-report
```

输出结果怎么看：

```text
# git branch 输出
* feat/weekly-tanglx-report    ← 你现在在这里
  main
  fix/login-error

# git branch -a 输出
* feat/weekly-tanglx-report
  main
  remotes/origin/HEAD -> origin/main
  remotes/origin/main
  remotes/origin/feat/weekly-tanglx-report
```

注意事项：`remotes/origin/xxx` 是远端追踪分支，不是你本地的工作分支。

### 4.3 分支命名规范

解释：规范的分支名让团队一眼看出这个分支是做什么的。

例子：

| 前缀 | 用途 | 例子 |
| --- | --- | --- |
| `feat/` | 新功能 | `feat/weekly-tanglx-report` |
| `fix/` | 修复 bug | `fix/login-timeout` |
| `docs/` | 文档修改 | `docs/update-readme` |
| `refactor/` | 重构代码 | `refactor/user-service` |
| `chore/` | 配置、依赖等杂项 | `chore/update-dependencies` |

注意事项：分支名只用小写字母、数字、短横线和斜杠，不要用空格和中文。

### 4.4 标准的新建分支操作流程

解释：每次开始新任务前，都应该从最新的 main 出发，新建功能分支。

命令：

```powershell
# 第一步：切换到 main
git switch main

# 第二步：拉取最新代码（确保从最新状态出发）
git pull

# 第三步：新建功能分支
git switch -c feat/weekly-tanglx-report

# 确认当前在新分支上
git branch
```

注意事项：如果不先 `git pull`，你的功能分支可能基于旧的 main，后面合并时更容易有冲突。

### 4.5 本地分支和远端分支的关系

解释：本地分支和远端分支是独立的，push 才能同步到远端，pull 才能从远端更新到本地。

例子：

```text
本地新建 feat/my-task                → 只有本地有
git push -u origin feat/my-task      → GitHub 上也有了
                                         本地 feat/my-task 追踪 origin/feat/my-task
队友 push 了 origin/feat/my-task     → 你本地需要 git pull 才能同步
```

注意事项：第一次 push 新分支时要用 `-u` 参数（见 08 章节），之后直接 `git push` 就可以了。

## 5. 和前后知识的关系

本节建立了分支的概念基础。下一节讲拉取代码时，会涉及"拉取哪个分支的内容"；08 节讲 push 时，会涉及"把当前分支推到远端"；PR 的本质就是"请求把我的功能分支合并进 main"。

## 6. 实战任务

在项目目录里完成：

```powershell
# 1. 确认当前在 main 分支
git switch main
git pull

# 2. 新建一个练习分支
git switch -c feat/practice-tanglx

# 3. 确认分支创建成功
git branch

# 4. 查看包含远端的所有分支
git branch -a

# 5. 切回 main
git switch main

# 6. 删除刚才的练习分支
git branch -d feat/practice-tanglx

# 7. 确认分支已删除
git branch
```

## 8. 常见误区

1. 误区：分支就是复制了一份文件。  
   解释：分支只是一个指针，不复制文件，切换分支是改变 HEAD 指向，Git 会更新工作区到对应版本。

2. 误区：分支建多了会很占空间。  
   解释：分支本身几乎不占空间，只是一个指针文件。分支上的 commit 数据才占空间。

3. 误区：在 feat 分支上改的东西，main 分支也能看到。  
   解释：切到 main 后，工作区会变成 main 的状态，feat 分支上的未合并改动不可见。

4. 误区：删除分支会删掉 commit。  
   解释：删除分支只是删掉指针，如果 commit 已经合并进其他分支，数据不会丢失。

## 9. 本节总结

分支让团队成员可以在互不干扰的平行环境里工作，改完再通过 PR 合并回主线。main 是受保护的稳定版本，所有改动都应该在 feature branch 上进行。新建分支前先从最新 main 出发，分支名使用 `类型/描述` 的规范格式。
