# 06 拉取与同步代码

## 1. 这一节要解决什么问题

这一节让你理解 `git fetch`、`git pull` 分别做了什么，为什么有时候 pull 之前要先确认工作区是干净的，以及 `git pull` 和 `git fetch` 的区别在哪里。

## 2. 基础概念解释

### git fetch

通俗解释：fetch 像邮件客户端的"检查新邮件"，只把远端的新内容下载到本地，但不打开邮件，也不把内容放进你的收件箱（工作区）。

技术解释：`git fetch` 从远端下载最新的提交和引用，更新本地的 `origin/xxx` 远端追踪分支，但不修改本地工作分支的任何文件。

例子：执行 `git fetch` 后，`origin/main` 更新了，但你本地的 `main` 分支还是原来的样子，工作区也没有任何变化。

### git pull

通俗解释：pull 像"检查新邮件并且自动把所有内容整合进你的工作文件夹"，一步完成下载和合并。

技术解释：`git pull` = `git fetch` + `git merge`（默认情况下），把远端最新内容下载后，自动合并到你当前所在的本地分支。

例子：执行 `git pull` 后，本地 main 分支更新到了远端最新状态，工作区文件也随之更新。

### Merge（合并）

通俗解释：merge 像把两条平行时间线的进度合并，把各自的改动都保留下来，合成一个新版本。

技术解释：merge 把两个分支的历史合并成一个，创建一个新的"合并提交"（如果有分叉的话），保留双方的所有 commit。

例子：`git pull` 内部就是 fetch + merge，把远端的新 commit 合并到你的本地分支上。

## 3. 为什么要学这个

团队协作中，队友随时可能 push 新代码。你如果不拉取最新代码就直接改，很容易和队友的改动产生冲突。养成"开始工作前先 pull"的习惯，是保持同步的关键。

## 4. 关键知识点

### 4.1 fetch 和 pull 的核心区别

解释：两者都从远端获取内容，区别在于是否自动合并到本地分支。

例子：

| 操作 | 下载远端内容 | 更新 origin/xxx | 改变本地分支 | 改变工作区 |
| --- | --- | --- | --- | --- |
| git fetch | ✅ | ✅ 更新 | ❌ | ❌ |
| git pull | ✅ | ✅ 更新 | ✅ 合并进来 | ✅ 文件更新 |

注意事项：fetch 更安全，可以先看看远端有什么变化再决定是否合并。pull 更方便，但如果本地有未提交改动可能触发冲突。

### 4.2 标准的每日开始工作流程

解释：每次开始工作前，先同步最新代码，再新建或切换到功能分支。

命令：

```powershell
# 第一步：切换到 main
git switch main

# 第二步：拉取最新代码
git pull

# 输出示例（有新内容时）：
# Updating 3a7b1d9..f2e8c1a
# Fast-forward
#  docs/weekly/2026-W25.md | 45 +++++++++++
#  1 file changed, 45 insertions(+)

# 输出示例（没有新内容时）：
# Already up to date.

# 第三步：新建功能分支
git switch -c feat/weekly-tanglx-w26
```

注意事项：`git pull` 之前最好确保工作区是干净的（git status 显示 nothing to commit），否则可能触发合并冲突。

### 4.3 先 fetch 再手动合并的方式

解释：想先看看远端有什么变化再决定操作，可以先 fetch。

命令：

```powershell
# 先下载远端内容，不合并
git fetch origin

# 查看远端比本地多了哪些 commit
git log main..origin/main --oneline

# 确认没问题后再合并
git merge origin/main
```

输出结果怎么看：`git log main..origin/main` 显示远端有但本地没有的提交列表，如果输出为空，说明本地和远端一致。

注意事项：平时日常工作直接用 `git pull` 就够了，fetch + merge 分开用适合有顾虑或有复杂情况时。

### 4.4 pull 时遇到未提交修改的处理

解释：如果工作区有未提交改动，pull 可能报错或触发冲突，有两种处理方式。

命令：

```powershell
# 情况1：把未提交改动暂存起来，pull 完再恢复
git stash          # 把当前未提交改动存起来
git pull           # 正常 pull
git stash pop      # 把改动取出来继续工作

# 情况2：先 commit 当前改动，再 pull
git add .
git commit -m "wip: 临时保存进度"
git pull
```

注意事项：`git stash` 是暂时保存工作进度的好工具，详见 `12_进阶概念补充`。

### 4.5 常见 pull 输出说明

解释：理解 pull 的输出能判断同步是否正常。

例子：

```text
# 情况1：fast-forward（最常见，直接向前推进）
Updating 3a7b1d9..f2e8c1a
Fast-forward
 docs/weekly/2026-W26.md | 45 ++++++
 1 file changed, 45 insertions(+)

含义：远端只有你没有的新提交，直接把本地 main 指针向前移动，不产生合并提交。

# 情况2：Already up to date
Already up to date.

含义：你本地和远端一模一样，不需要做任何事。

# 情况3：合并冲突（需要手动解决）
Auto-merging docs/weekly/2026-W26.md
CONFLICT (content): Merge conflict in docs/weekly/2026-W26.md
Automatic merge failed; fix conflicts and then commit the result.

含义：你和队友修改了同一个文件的同一个位置，需要手动解决冲突。见 10_常见报错。
```

注意事项：看到 CONFLICT 不要慌，这是正常情况，冲突解决后 commit 即可。

## 8. 常见误区

1. 误区：fetch 了就等于 pull 了，代码更新了。  
   解释：fetch 只更新 `origin/main` 这个远端追踪指针，你的本地 main 分支和工作区文件没有变化。

2. 误区：pull 之前不需要切换到 main。  
   解释：`git pull` 更新的是你当前所在分支，如果你在功能分支上 pull，更新的是功能分支，不是 main。

3. 误区：每次 pull 都会产生合并提交。  
   解释：如果远端只有你没有的提交（fast-forward 情况），不会产生额外的合并提交，历史是一条直线。

4. 误区：pull 失败了数据就丢了。  
   解释：pull 失败（比如冲突）时，Git 会停下来等你处理，不会自动丢弃任何数据。

## 9. 本节总结

`git fetch` 只下载不合并，`git pull` 下载并合并。日常工作的节奏是：开始工作前 pull 最新代码 → 新建功能分支 → 工作 → commit → push。如果工作区有未提交改动，pull 之前先 stash 或先 commit，避免冲突。
