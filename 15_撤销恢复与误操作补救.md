# 15 撤销恢复与误操作补救

## 1. 这一节要解决什么问题

这一节专门解决“我改错了、add 错了、commit 错了、push 后发现错了、提交到错分支了”这些真实工作里很常见的问题。

学完这一节，你应该能先判断错误发生在哪个阶段，再选择相对安全的补救方式，而不是一紧张就 `git reset --hard`。

## 2. 基础概念解释

### git restore

通俗解释：`git restore` 像把文件恢复到某个已知版本，可以用来撤销工作区修改。

技术解释：`git restore 文件名` 默认把工作区文件恢复到暂存区或最近一次 commit 的状态。

例子：你改坏了 `README.md`，还没 add，可以用 `git restore README.md` 丢弃这次修改。

### git restore --staged

通俗解释：把已经放进暂存区的文件拿出来，但文件内容还留在工作区。

技术解释：`git restore --staged 文件名` 会取消暂存，不会删除实际修改。

例子：你不小心 `git add .`，可以用 `git restore --staged .` 把文件从暂存区移出。

### git reset

通俗解释：`git reset` 像移动当前分支指针，不同参数决定是否保留修改。

技术解释：`reset` 会改变当前分支指向的 commit，并根据 `--soft`、`--mixed`、`--hard` 影响暂存区和工作区。

例子：最近一次 commit 还没 push，想拆开重提，可以用 `git reset --soft HEAD~1`。

### git revert

通俗解释：`git revert` 是新增一个“反向提交”，用新 commit 抵消旧 commit 的影响。

技术解释：`revert` 不改写历史，适合撤销已经 push 到共享分支上的提交。

例子：某个 bug commit 已经合进 main，不要 reset main，应该 `git revert commit_id`。

### git reflog

通俗解释：`reflog` 像 Git 的本地操作记录，可以帮你找回刚刚“看不见”的 commit。

技术解释：`git reflog` 记录 HEAD 和分支引用在本地的移动历史，可用于恢复误 reset、误切换后丢失的提交。

例子：你 reset 后发现错了，可以从 `git reflog` 找到 reset 前的 commit hash。

## 3. 为什么要学这个

Git 真正让人害怕的不是正常提交，而是出错后的处理。很多数据丢失不是 Git 导致的，而是误用了破坏性命令。掌握撤销恢复的判断方法，可以让你在出错时稳住节奏，先保住数据，再修正历史。

## 4. 关键知识点

### 4.1 先判断错误发生在哪一层

解释：不同阶段的错误，用不同命令处理。

| 错误位置 | 典型情况 | 推荐命令 |
| --- | --- | --- |
| 工作区 | 改了文件，还没 add | `git restore 文件名` |
| 暂存区 | add 错了，还没 commit | `git restore --staged 文件名` |
| 本地 commit | commit 错了，还没 push | `git reset --soft HEAD~1` 或 `git commit --amend` |
| 远端历史 | commit 已经 push | 优先 `git revert commit_id` |
| 找不到提交 | reset 或 detached HEAD 后 commit 不见了 | `git reflog` |

注意事项：先运行 `git status` 和 `git log --oneline -5`，再决定用什么命令。

### 4.2 撤销工作区修改

解释：适用于“文件改坏了，且还没 add”的情况。

命令：

```powershell
# 查看改了什么
git diff README.md

# 确认不要这些修改后，丢弃工作区修改
git restore README.md
```

丢弃多个文件：

```powershell
# 谨慎：丢弃所有已追踪文件的工作区修改
git restore .
```

注意事项：`git restore 文件名` 会丢弃未提交修改，确认不要了再执行。

### 4.3 取消暂存但保留修改

解释：适用于“add 错了，但修改本身还想保留”的情况。

命令：

```powershell
# 取消暂存单个文件
git restore --staged README.md

# 取消暂存所有文件
git restore --staged .
```

执行前后区别：

```text
执行前：Changes to be committed
执行后：Changes not staged for commit
```

注意事项：这条命令不丢文件内容，只是把文件从暂存区拿回工作区。

### 4.4 修改最近一次 commit

解释：适用于“刚 commit 完，发现 message 写错了，或漏 add 了一个文件，并且还没 push”。

只改 commit message：

```powershell
git commit --amend -m "docs: 修正 Git 学习路线说明"
```

漏文件后补进最近一次 commit：

```powershell
git add docs/git-github-learning/README.md
git commit --amend --no-edit
```

注意事项：`amend` 会改写最近一次 commit。已经 push 到共享分支后，不要随便 amend。

### 4.5 撤销最近一次本地 commit

解释：适用于“commit 错了，还没 push，想重新整理”。

保留所有修改到暂存区：

```powershell
git reset --soft HEAD~1
```

保留修改到工作区，但取消暂存：

```powershell
git reset --mixed HEAD~1
```

三种 reset 区别：

| 命令 | commit 是否撤销 | 暂存区 | 工作区文件 |
| --- | --- | --- | --- |
| `git reset --soft HEAD~1` | 是 | 保留 | 保留 |
| `git reset --mixed HEAD~1` | 是 | 取消暂存 | 保留 |
| `git reset --hard HEAD~1` | 是 | 丢弃 | 丢弃 |

注意事项：新手优先用 `--soft` 或 `--mixed`，不要把 `--hard` 当成普通撤销命令。

### 4.6 撤销已经 push 的提交

解释：已经 push 到远端，尤其是 main 或团队共享分支，推荐用 `revert`，不要改写历史。

命令：

```powershell
# 找到要撤销的 commit
git log --oneline

# 新增一个反向提交
git revert abc1234

# 推送这个反向提交
git push
```

结果：

```text
旧 commit 仍然保留在历史里
新增一个 revert commit 抵消旧改动
团队其他人 pull 时不会被破坏
```

注意事项：如果 revert 时出现冲突，按 10 节的冲突处理方式解决后继续提交。

### 4.7 用 reflog 找回误操作前的位置

解释：适用于“我 reset 了、切来切去之后找不到刚才的 commit 了”。

命令：

```powershell
# 查看本地 HEAD 移动历史
git reflog
```

输出示例：

```text
f2e8c1a HEAD@{0}: reset: moving to HEAD~1
3a7b1d9 HEAD@{1}: commit: docs: 添加 Git 补充篇
```

恢复方式：

```powershell
# 方式一：新建一个救援分支
git switch -c rescue-lost-commit 3a7b1d9

# 方式二：把这个 commit 拿回当前分支
git switch 正确分支名
git cherry-pick 3a7b1d9
```

注意事项：`reflog` 是本地记录，不是远端记录。越早发现越容易救。

### 4.8 提交到错分支怎么办

解释：这是新手和老手都会遇到的问题。先保住 commit，再把它移到正确分支。

场景：你本来应该在 `feat/git-docs` 上提交，却提交到了 `main`。

处理流程：

```powershell
# 1. 记住错误提交的 hash
git log --oneline -3

# 假设错误 commit 是 abc1234

# 2. 切到正确分支
git switch feat/git-docs

# 3. 把那个 commit 拿过来
git cherry-pick abc1234

# 4. 回到 main
git switch main

# 5. 确认工作区干净，并且错误 commit 已经复制到正确分支
git status

# 6. 如果 main 上的错误 commit 还没 push，可以从 main 移除它
git reset --hard HEAD~1
```

注意事项：这里的 `reset --hard` 只适合“错误 commit 还没 push、工作区干净、commit 已经 cherry-pick 到正确分支”的场景。如果错误 commit 已经 push 到共享 main，不要自己 reset main，先和团队沟通，通常用 revert。

## 8. 常见误区

1. 误区：撤销就用 `git reset --hard`。  
   解释：大多数撤销都不需要 `--hard`，它会丢弃工作区修改。

2. 误区：已经 push 的 commit 可以随便 amend。  
   解释：amend 会改写 commit hash，团队共享历史会被影响。

3. 误区：`git revert` 会删除历史。  
   解释：revert 不删除历史，而是新增一个反向提交。

4. 误区：commit 丢了就一定找不回。  
   解释：本地短时间内通常可以从 `git reflog` 找到。

## 9. 本节总结

撤销恢复的核心是先判断错误发生在哪一层：工作区用 `restore`，暂存区用 `restore --staged`，未 push 的本地 commit 可以 `amend` 或 `reset --soft`，已 push 的共享提交优先 `revert`，找不到 commit 时查 `reflog`。不要把 `reset --hard` 当成默认补救方案。
