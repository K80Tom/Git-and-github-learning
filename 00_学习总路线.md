# 00 学习总路线

## 1. 这一节要解决什么问题

这一节帮你看清"Git 与 GitHub 协作"从配置、克隆、分支、提交、推送到 Pull Request 的完整路线，知道先学什么、后学什么、每天怎么安排，以及遇到问题怎么定位。

## 2. 基础概念解释

### Git

通俗解释：Git 像一本自动存档的日记本，装在你自己电脑上，记录每次修改了什么、谁改的、什么时候改的。

技术解释：Git 是一个分布式版本控制系统，在本地追踪文件变更历史，支持分支、合并、回滚等操作，不依赖网络。

例子：你改了一段代码，发现改坏了，Git 让你随时回到之前的任意版本，不依靠 Ctrl+Z。

### GitHub

通俗解释：GitHub 像班级的共享网盘，所有人把自己的日记本同步到这里，方便一起协作。

技术解释：GitHub 是基于 Git 的远端代码托管平台，提供仓库存储、Pull Request、Issue、Actions 等协作功能，是目前最主流的开源和团队协作平台。

例子：老师建了一个仓库放在 GitHub，你 clone 下来，改完内容 push 回去，再发 PR 让老师 review。

### 版本控制

通俗解释：版本控制像文档的历史记录，每次存档都有快照，随时可以翻看或恢复。

技术解释：版本控制系统追踪文件集合随时间的变化，使团队成员可以并行工作、合并修改、追溯历史。

例子：你和同学同时修改不同文件，Git 帮你自动合并，不用手工对比两份文档。

## 3. 为什么要学这个

如果你不理解 Git 和 GitHub，你只能照着命令一行一行敲，出错就不知道发生了什么。掌握这套工具后，你能和团队安全协作，能看懂项目历史，能独立解决常见问题，并且知道哪些命令可能导致数据丢失。

### 4.3 7 天快速入门版

解释：适合先建立全局认知，目标是能完整走一遍"克隆→改文件→提交→推送→开 PR"的流程。

| 天数 | 学习内容 | 阅读文件 | 动手任务 | 时间 | 达标标准 | 学不懂怎么补救 |
| --- | --- | --- | --- | --- | --- | --- |
| 第 1 天 | Git 是什么，GitHub 是什么 | 00、01 | 用 100 字解释两者区别 | 1.5h | 能说出 Git 是本地工具、GitHub 是远端平台 | 只看通俗解释和例子部分 |
| 第 2 天 | 配置 Git 和 SSH | 02 | 完成 user.name / user.email 配置，生成 SSH Key 并添加到 GitHub | 2h | 能 `ssh -T git@github.com` 得到成功响应 | 只做 HTTPS 方式，SSH 下周再补 |
| 第 3 天 | 克隆仓库，理解 origin | 03 | 把团队仓库 clone 到本地，运行 `git remote -v` | 1.5h | 看到 origin 地址 | 直接复制老师给的命令 |
| 第 4 天 | 三个区域 + 分支 | 04、05 | 在本地新建分支 `feat/test-branch` | 2h | 能解释 working tree 和 staging area 的区别 | 画一张纸质的三区域图 |
| 第 5 天 | 拉取 + 提交 | 06、07 | 改一个文件，完成 add → commit，写规范 commit message | 2h | 能用 `git log --oneline` 看到自己的提交 | 先用 `git add 文件名` 而不是 `git add .` |
| 第 6 天 | 推送 + PR | 08 | push 分支到 GitHub，在网页开一个 PR | 2h | 能在 GitHub 上看到自己的 PR | 不懂 conflict 先跳过 |
| 第 7 天 | 看懂状态 + 报错 | 09、10 | 认识 git status 五种状态，查阅一条报错 | 2h | 能判断"工作区干不干净" | 遇到报错先找 10 节对应描述 |

注意事项：7 天版的目标不是掌握所有命令，而是跑通一次完整协作流程。

### 4.6 学习方法建议

解释：学 Git 要用"每次操作后立刻观察"的方式，不要只看文档不动手。

例子：

```text
每次操作前：先 git status 看清楚当前状态
每次操作后：再 git status 或 git log 确认变化
遇到不认识的输出：先不要乱操作，查文档再说
遇到报错：复制报错关键词，查 10_常见报错
遇到危险命令：先看 11_危险命令章节，确认能不能操作
想知道命令做了什么：先用 --help 或查文档
```

注意事项：Git 命令很多是不可逆的，不懂的命令先不要执行，查清楚再操作。

## 8. 常见误区

1. 误区：Git 就是备份工具。  
   解释：Git 的核心价值是追踪变更历史和支持多人并行协作，备份只是附带效果。

2. 误区：GitHub 就是 Git。  
   解释：Git 是本地工具，GitHub 是远端平台，两者可以分开用。

3. 误区：会了几条命令就会 Git 了。  
   解释：命令本身很简单，难的是理解背后的协作模型：工作区/暂存区/仓库、本地分支/远端分支、fetch/pull/push 的差异。

4. 误区：出错了随便用 `git reset --hard` 就能解决。  
   解释：这个命令会丢弃所有未提交修改，新手不能随便用。

## 9. 本节总结

Git 和 GitHub 学习要按"先理解模型，再掌握命令"的顺序推进。7 天计划适合建立第一印象，14 天计划适合系统掌握，30 天计划适合把习惯固化到真实项目中。`00` 到 `13` 是日常协作主线，`14` 到 `18` 是补充篇。最重要的原则是：每次操作前先看状态，不懂的命令先查再用。
