# 04 三个区域与工作流程

## 1. 这一节要解决什么问题

这一节让你真正理解 Git 里最核心的三个区域：Working Tree（工作区）、Staging Area（暂存区）、Repository（本地仓库），以及文件是怎么从"你改了一下"一步一步到"有历史记录"的。

## 2. 基础概念解释

### Working Tree（工作区）

通俗解释：工作区就是你用编辑器看到和修改的那些文件，是你实际工作的地方。

技术解释：Working Tree 是 Git 仓库目录下，除了 `.git` 目录之外所有文件所在的区域，代表当前磁盘上文件的实际状态。

例子：你用 VS Code 打开 `C:\Users\Firebat\zhongwen-shortdrama-context`，你看到和编辑的所有文件，都在工作区里。

### Staging Area（暂存区 / Index）

通俗解释：暂存区像一个"待寄快递箱"，你把想要一起提交的文件先放进去，打包好了再统一寄出去（commit）。

技术解释：Staging Area（也叫 Index）是介于工作区和本地仓库之间的缓冲层，`git add` 把工作区的改动放入暂存区，`git commit` 把暂存区的内容打包成一个提交。

例子：你改了 3 个文件，但只想提交其中 2 个，可以只 `git add` 那 2 个文件，然后 commit，第 3 个文件留在工作区继续改。

### Repository（本地仓库）

通俗解释：本地仓库像一本存档完整的日记本，每次 commit 都是一条不可删改的日记，永久保留。

技术解释：Repository 是 `.git` 目录里存放的提交历史、分支、标签等所有版本控制数据的集合，commit 之后的内容就进入这里。

例子：`git log` 显示的每一条记录，都是存在本地仓库里的一个快照。

### Commit（提交）

通俗解释：commit 像给工作存档，把你这次改动和一条说明文字一起打包，附上时间戳和作者，永久存在历史里。

技术解释：commit 是 Git 仓库里的一个不可变快照，包含这次提交的文件状态、父提交的引用、作者信息、时间和提交信息。

例子：一个 commit 有唯一的 hash 值（如 `3a7b1d9`），可以随时回到这个状态。

## 3. 为什么要学这个

很多人不理解为什么"改了文件还要 add，add 了还要 commit"，觉得多此一举。实际上这三步分别解决三个问题：工作区让你自由改文件；暂存区让你精确控制"这次提交包含哪些变化"；commit 让变化永久记录。跳过任何一步都会出问题。

## 4. 关键知识点

### 4.1 三个区域的关系图

解释：文件的每次状态变化，都对应一个 Git 操作。

例子：

```text
┌──────────────────────────────────────────────────────────┐
│                         你的电脑                          │
│                                                          │
│  ┌─────────────┐   git add   ┌──────────────┐           │
│  │  Working    │ ──────────► │   Staging    │           │
│  │  Tree       │ ◄────────── │   Area       │           │
│  │  工作区     │  git restore │   暂存区     │           │
│  └─────────────┘             └──────┬───────┘           │
│                                     │ git commit         │
│                                     ▼                    │
│                              ┌──────────────┐           │
│                              │  Repository  │           │
│                              │  本地仓库    │           │
│                              └──────────────┘           │
└──────────────────────────────────────────────────────────┘
         ↕  git push / git pull / git fetch
┌──────────────────────────────────────────────────────────┐
│                    GitHub（远端）                         │
└──────────────────────────────────────────────────────────┘
```

注意事项：push/pull 只在本地仓库和远端之间操作，不直接涉及工作区和暂存区。

### 4.2 文件从新建到提交的完整生命周期

解释：一个文件从创建到有历史记录，要经过三个状态转变。

例子：

```powershell
# 第一步：新建一个文件
# 此时文件状态：Untracked（未追踪，Git 不认识它）
New-Item docs\weekly\2026-W26.md

# 查看状态
git status
# 输出：Untracked files: docs/weekly/2026-W26.md

# 第二步：git add 把文件放入暂存区
# 此时文件状态：Staged（已暂存）
git add docs\weekly\2026-W26.md

# 查看状态
git status
# 输出：Changes to be committed: new file: docs/weekly/2026-W26.md

# 第三步：git commit 把暂存区的内容打包成提交
# 此时文件进入本地仓库
git commit -m "docs: 新增 2026-W26 周报"

# 查看状态
git status
# 输出：nothing to commit, working tree clean

# 查看历史
git log --oneline -3
```

注意事项：commit 之后文件还在工作区，只是工作区和本地仓库保持一致了，所以 git status 显示"干净"。

### 4.3 为什么有暂存区（不直接 commit 所有改动）

解释：暂存区让你可以精确控制这次 commit 包含哪些改动，而不是必须把所有修改都一起提交。

例子：

```text
情景：你同时修改了 3 个文件
  - docs/weekly/2026-W26.md    ← 这周的周报，写完了
  - src/main.py                ← 正在写的功能，还没完成
  - README.md                  ← 随手改了一个错别字

理想方案：把周报和 README 分两个 commit 提交，src/main.py 先不提交

操作：
git add docs/weekly/2026-W26.md
git commit -m "docs: 新增 2026-W26 周报"

git add README.md
git commit -m "fix: 修正 README 错别字"

# src/main.py 继续在工作区，不影响上面两个 commit
```

注意事项：`git add .` 会把所有修改都放入暂存区，新手容易一不小心把不想提交的文件也加进去，建议优先用 `git add 具体文件名`。

### 4.4 撤销暂存区的文件（把文件从暂存区取出来）

解释：如果不小心 add 了不想提交的文件，可以把它从暂存区取出，放回工作区。

命令：

```powershell
# 把文件从暂存区取出（不影响工作区的实际内容）
git restore --staged 文件名

# 例子：
git restore --staged src/main.py
```

输出结果怎么看：运行后该文件从 "Changes to be committed" 移回 "Changes not staged"，工作区内容不变。

注意事项：`git restore --staged` 是安全的，只影响暂存区，不会丢弃工作区的修改。

### 4.5 查看每个区域的改动

解释：不同命令可以查看不同区域的差异。

命令：

```powershell
# 查看工作区 vs 暂存区的差异（改了但还没 add 的内容）
git diff

# 查看暂存区 vs 最近一次 commit 的差异（已 add 但还没 commit 的内容）
git diff --staged

# 查看工作区 vs 最近一次 commit 的总差异
git diff HEAD
```

注意事项：`git diff` 什么都不显示，不代表"没有修改"，可能是修改都已经 add 进暂存区了。这时要用 `git diff --staged` 才能看到。

## 5. 和前后知识的关系

本节是理解 `git add`、`git commit`、`git restore` 的核心基础。后面讲 git status 时，显示的每种状态都对应文件在哪个区域、处于什么状态。push/pull 只涉及本地仓库和远端，不直接操作工作区和暂存区。

## 6. 实战任务

在项目目录里完成：

```powershell
# 1. 新建一个测试文件
New-Item test-staging.txt

# 2. 查看状态（应该看到 Untracked）
git status

# 3. git add 这个文件
git add test-staging.txt

# 4. 查看状态（应该看到 staged）
git status

# 5. 从暂存区取出（不提交这个测试文件）
git restore --staged test-staging.txt

# 6. 查看状态（回到 Untracked）
git status

# 7. 删除这个测试文件（清理）
Remove-Item test-staging.txt
```

观察每一步 git status 的变化，记录下三种状态的输出是什么样的。

## 8. 常见误区

1. 误区：git add 就是"保存文件"。  
   解释：git add 是"把文件改动放入暂存区准备提交"，不是保存文件。文件在磁盘上编辑时就已经保存了（Ctrl+S）。

2. 误区：必须 git add . 才能 commit。  
   解释：只有已经 git add 的文件才会进入 commit，可以只 add 你想提交的文件，其他文件不影响。

3. 误区：commit 之后文件消失了。  
   解释：commit 只是给当前状态创建了一个历史快照，文件仍然在工作区，没有任何变化。

4. 误区：push 就是 commit。  
   解释：commit 是把改动保存到本地历史，push 是把本地历史上传到 GitHub，是两步独立操作。

## 9. 本节总结

Git 的三个区域：工作区是你修改文件的地方，暂存区是你精确打包"这次要提交什么"的地方，本地仓库是永久存档历史的地方。流程是：改文件 → `git add` → `git commit`，缺一不可。理解了这三层，git status 里的每种状态才会有意义。
