# 01 Git 与 GitHub 是什么

## 1. 这一节要解决什么问题

这一节让你真正理解 Git 和 GitHub 各自是什么、它们的分工是什么，以及为什么团队项目需要两者配合，而不是混为一谈。

## 2. 基础概念解释

### Git

通俗解释：Git 是装在你电脑上的"本地版本日记本"，每次提交就像在日记本上盖一个时间戳，记录"谁在什么时候改了什么"。

技术解释：Git 是一个开源的分布式版本控制系统，完全运行在本地，不需要网络也能追踪文件变更、创建分支、回滚历史。

例子：你改了一段代码，发现改错了，但已经保存了。Git 可以让你回到任意一个之前的版本，而不依靠 Ctrl+Z。

### GitHub

通俗解释：GitHub 是互联网上的"共享版本日记柜"，让团队所有人都可以把各自的本地日记本同步到同一个地方，看到彼此的进度。

技术解释：GitHub 是一个基于 Git 的远端代码托管平台，提供仓库存储、分支保护、Pull Request、Issue、代码审查、Actions（自动化）等协作功能。

例子：老师建了一个 GitHub 仓库，你把它 clone 到本地，改完 push 回去，再发一个 PR 请老师 review，就完成了一次完整协作。

### 仓库（Repository / Repo）

通俗解释：仓库是一个文件夹加上这个文件夹里所有文件的完整修改历史。

技术解释：Git 仓库是一个包含 `.git` 隐藏目录的文件夹，`.git` 里存放了所有提交历史、配置和引用信息。

例子：`C:\Users\Firebat\zhongwen-shortdrama-context` 如果里面有 `.git` 目录，它就是一个本地 Git 仓库。

### 本地与远端

通俗解释：本地是你自己电脑上的副本，远端是 GitHub 上的副本。它们不会自动同步，需要手动 push 或 pull。

技术解释：本地仓库（local repository）存在于你的磁盘上，远端仓库（remote repository）存在于 GitHub 等平台上。两者通过 push/pull/fetch 交换数据。

例子：你在本地写了三个 commit，但没有 push，GitHub 上是看不到的。同样，队友 push 了新内容，你本地也看不到，需要 pull 才能同步。

## 3. 为什么要学这个

如果你把 Git 和 GitHub 混为一谈，就很难理解"为什么改了文件 GitHub 上没变化""为什么 push 之前要先 add 再 commit"这类问题。分清两者的边界是后面所有操作的基础。

## 4. 关键知识点

### 4.1 Git 和 GitHub 的核心分工

解释：Git 负责本地的版本追踪，GitHub 负责远端的存储和协作。两者角色不同，但配合使用。

例子：

```text
你的电脑（本地）
├── Git 追踪文件变化
├── Git 管理分支
└── Git 记录 commit 历史
         ↕  push / pull / fetch
GitHub（远端）
├── 存储代码和历史
├── 展示 PR 和 Review
└── 保护 main 分支
```

注意事项：GitHub 只是众多远端平台之一，类似的还有 GitLab、Gitee、Bitbucket。但用法相同，Git 本身不绑定任何平台。

### 4.2 为什么不能只用 GitHub 不用 Git

解释：GitHub 只是存代码的地方，代码的修改、版本切换、分支创建都发生在本地 Git 上。没有 Git，你没法跟踪本地变化。

例子：就像你不能只有共享网盘而没有本地文档编辑器，网盘存文件，但文件还是要在本地改。

注意事项：GitHub 网页上虽然也能直接编辑文件，但正常团队开发都在本地完成，GitHub 只作为共享中心。

### 4.3 为什么不能只用 Git 不用 GitHub

解释：只用本地 Git，当你的电脑坏了，所有历史都丢了。而且团队成员没法访问你的本地仓库。

例子：就像只有草稿本没有共享网盘，你改好了队友也看不到，更没法审查。

注意事项：对个人项目来说，只用本地 Git 是可以的。但团队协作必须有远端仓库。

### 4.4 一次完整协作流程的样子

解释：了解完整流程能帮你理解每个命令存在的意义。

例子：

```text
1. 老师在 GitHub 上创建仓库（远端已有代码）
2. 你 clone 到本地（本地有了一份副本）
3. 你新建一个分支（不影响 main）
4. 你修改文件（本地工作区变化）
5. git add → git commit（保存到本地历史）
6. git push（上传到 GitHub）
7. 在 GitHub 发起 Pull Request（请老师 review）
8. 老师 review 并 merge（代码合进 main）
9. 你 git pull 更新本地（同步最新 main）
```

注意事项：这个流程的每一步都有意义，不能随意跳过。跳过会导致冲突或权限问题。

### 4.5 Git 的版本控制具体记录什么

解释：Git 不只记录"保存了文件"，它记录每次提交时哪些行被增加、删除、修改，以及谁提交的、什么时候提交的。

例子：

```text
commit 3a7b1d9
Author: Firebat <firebat@example.com>
Date:   2026-06-26

    docs: 添加第三周周报

    新增文件 docs/weekly/2026-W26.md
    修改 README.md 第 5 行
```

注意事项：这就是为什么 Git 需要配置用户名和邮箱，每条 commit 记录都会带上提交者身份。

## 8. 常见误区

1. 误区：GitHub 就是 Git。  
   解释：Git 是版本控制工具，GitHub 是托管平台，Git 可以不用 GitHub，但 GitHub 依赖 Git。

2. 误区：我在 GitHub 网页上改了文件，本地就同步了。  
   解释：本地和远端不会自动同步，需要手动 pull。

3. 误区：push 了代码就相当于"保存"了。  
   解释：push 之前必须先 commit，commit 之前还要 add。push 只是"上传"，commit 才是"保存版本"。

4. 误区：删掉 `.git` 文件夹不影响代码。  
   解释：`.git` 里存放了所有历史记录和版本信息，删掉之后这个目录就不再是 Git 仓库了，所有历史也消失了。

## 9. 本节总结

Git 是本地的版本追踪工具，GitHub 是远端的协作平台。两者配合完成完整的团队协作流程：本地 Git 追踪变化、提交历史，GitHub 存储共享版本、管理 PR 和权限。理解这个分工，是后面所有操作的基础。
