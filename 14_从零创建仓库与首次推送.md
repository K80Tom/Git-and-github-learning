# 14 从零创建仓库与首次推送

## 1. 这一节要解决什么问题

前面的内容主要围绕“团队已经有一个 GitHub 仓库，你 clone 下来再协作”。这一节补齐另一个常见场景：你自己从零创建一个项目，把本地代码第一次推送到 GitHub。

学完这一节，你应该能判断什么时候用 `git clone`，什么时候用 `git init`，并能完成“本地项目 → GitHub 新仓库 → 首次 push”的完整流程。

## 2. 基础概念解释

### git init

通俗解释：`git init` 像是在一个普通文件夹里放进一本版本记录本，从这一刻开始 Git 才能追踪这个文件夹里的变化。

技术解释：`git init` 会在当前目录创建 `.git` 文件夹，使当前目录成为一个 Git 仓库。

例子：你新建了 `my-ai-demo` 文件夹，里面已经有代码，但还不是 Git 仓库，执行 `git init` 后才能 `git add` 和 `git commit`。

### GitHub 新建仓库

通俗解释：GitHub 新建仓库就是在云端创建一个空项目位置，等你把本地代码上传过去。

技术解释：GitHub 仓库是远端仓库，通常会作为本地仓库的 `origin` remote。

例子：你在 GitHub 上创建 `my-ai-demo` 仓库，GitHub 会给你一个 SSH 地址：`git@github.com:用户名/my-ai-demo.git`。

### git remote add

通俗解释：`git remote add` 是告诉本地仓库：“以后上传下载，就去这个 GitHub 地址。”

技术解释：`git remote add origin 地址` 会新增一个名为 `origin` 的远端引用，指向 GitHub 仓库。

例子：`git remote add origin git@github.com:Firebat/my-ai-demo.git`

### 首次 push

通俗解释：首次 push 是把本地已有的提交第一次上传到 GitHub，并建立本地分支和远端分支的追踪关系。

技术解释：`git push -u origin main` 会把本地 `main` 分支推送到远端 `origin/main`，并设置 upstream。

例子：第一次成功后，之后在 `main` 上直接 `git push` 即可。

## 3. 为什么要学这个

如果你只会 clone，就只能参与别人已经建好的项目。会从零创建仓库后，你才能把自己的练习项目、作业、作品集、课程代码托管到 GitHub，也能理解为什么有些人误用 `git init` 后会遇到 `unrelated histories` 这类问题。

## 4. 关键知识点

### 4.1 什么时候用 clone，什么时候用 init

解释：已有远端仓库时优先 clone；本地先有项目、远端还没有时才用 init。

| 场景 | 推荐操作 | 原因 |
| --- | --- | --- |
| 老师或团队已经给了 GitHub 仓库地址 | `git clone 仓库地址` | 自动下载代码并设置 remote |
| GitHub 上已有项目，你想参与协作 | `git clone 仓库地址` | 保留完整历史和分支关系 |
| 你本地已经写了一个新项目，GitHub 上还没有仓库 | `git init` + 新建 GitHub 仓库 | 把本地项目变成 Git 仓库后再上传 |
| 当前目录不是项目根目录，只是随便一个文件夹 | 不要急着 `git init` | 可能把错误目录变成仓库 |

注意事项：团队项目中，如果远端仓库已经存在，通常不要自己 `git init` 后再 `remote add`，应该直接 clone。

### 4.2 本地先有项目的标准流程

解释：这是最常见的“我已经写了一点代码，现在想上传 GitHub”的流程。

命令：

```powershell
# 1. 进入项目根目录
cd D:\projects\my-ai-demo

# 2. 初始化 Git 仓库
git init

# 3. 查看状态
git status

# 4. 添加必要文件
git add README.md .gitignore src

# 5. 提交第一个版本
git commit -m "chore: 初始化项目"

# 6. 确认当前分支名
git branch
```

如果当前分支不是 `main`，可以改名：

```powershell
git branch -M main
```

注意事项：第一次提交前建议先准备好 `.gitignore`，避免把 `.env`、缓存文件、依赖目录提交进去。

### 4.3 在 GitHub 创建空仓库

解释：如果本地已经有 README 或代码，GitHub 新建仓库时建议不要勾选自动生成 README，避免两边都有初始化提交。

步骤：

```text
1. 打开 GitHub
2. 点击右上角 + → New repository
3. 填写 Repository name，例如 my-ai-demo
4. 选择 Public 或 Private
5. 如果本地已经有 README/.gitignore/License：
   不勾选 Add a README file
   不选择 .gitignore 模板
   不选择 License
6. 点击 Create repository
7. 复制 SSH 地址
```

注意事项：本地已有提交时，GitHub 端保持空仓库最省事。

### 4.4 添加 remote 并首次推送

解释：本地仓库和 GitHub 仓库是两份独立仓库，需要用 remote 建立连接。

命令：

```powershell
# 添加远端地址
git remote add origin git@github.com:用户名/my-ai-demo.git

# 检查 remote 是否正确
git remote -v

# 首次推送 main 分支
git push -u origin main
```

输出结果怎么看：

```text
To github.com:用户名/my-ai-demo.git
 * [new branch]      main -> main
Branch 'main' set up to track remote branch 'main' from 'origin'.
```

含义：

```text
[new branch] 说明 GitHub 上新建了 main 分支
set up to track 说明本地 main 和 origin/main 建立了追踪关系
```

注意事项：如果提示 `Permission denied (publickey)`，回到 02 节检查 SSH 配置。

### 4.5 GitHub 先有 README 的处理

解释：如果你在 GitHub 新建仓库时勾选了 README，而本地也已经有 commit，就会出现两边历史不同的情况。

推荐处理方式：

```powershell
# 如果远端只有 README，且你确认要保留远端 README
git pull origin main --allow-unrelated-histories

# 处理可能出现的冲突
git status
git add README.md
git commit -m "merge: 合并远端初始化文件"

# 再推送
git push -u origin main
```

更推荐的预防方式：

```text
本地已有项目 → GitHub 新建空仓库，不勾选 README
GitHub 先建仓库并勾选 README → 先 clone，再把代码复制进去
```

注意事项：`--allow-unrelated-histories` 只适合初始化时的一次性场景，不要当成日常命令。

### 4.6 常见初始化文件

解释：一个干净的新仓库通常至少包含 README 和 .gitignore。

常见文件：

| 文件 | 作用 |
| --- | --- |
| `README.md` | 说明项目是什么、怎么运行、怎么使用 |
| `.gitignore` | 忽略不该提交的文件 |
| `LICENSE` | 声明开源协议，个人私有项目可暂时不加 |
| `.env.example` | 提供环境变量示例，不包含真实密钥 |

`.gitignore` 示例：

```gitignore
# Python
__pycache__/
.venv/

# Node.js
node_modules/
dist/

# Secrets
.env

# Editor
.vscode/
.idea/
```

注意事项：不要提交真实的 `.env`、API Key、数据库密码。

## 5. 和前后知识的关系

这一节补齐了 03 节没有展开的“从零建仓库”场景。03 节适合已有远端仓库时使用；本节适合本地先有项目时使用。遇到初始化冲突时，可以回看 10 节的 `unrelated histories`。

## 6. 实战任务

用一个练习目录完整走一遍首次推送：

```powershell
mkdir my-git-init-practice
cd my-git-init-practice

# 新建 README.md 后，初始化仓库
git init
git add README.md
git commit -m "chore: 初始化练习仓库"

# 在 GitHub 创建空仓库后，替换成你自己的地址
git remote add origin git@github.com:用户名/my-git-init-practice.git
git push -u origin main
```

完成后，在 GitHub 页面确认 README 能正常显示。

## 8. 常见误区

1. 误区：任何时候新项目都先 `git init`。  
   解释：如果远端仓库已经存在，应该 clone，否则容易制造两份不相关历史。

2. 误区：GitHub 上勾选 README 更方便。  
   解释：如果本地已经有 commit，远端自动 README 会制造额外初始化提交，首次 push 更容易报错。

3. 误区：remote 名字必须叫 origin。  
   解释：origin 只是约定俗成的默认名字，可以换，但新手阶段建议保持 origin。

4. 误区：`.gitignore` 后加也一样。  
   解释：已经被 Git 追踪的文件，后来写进 `.gitignore` 不会自动停止追踪。

## 9. 本节总结

从零创建仓库的推荐流程是：本地项目根目录 `git init` → 准备 `.gitignore` 和 README → `git add` → `git commit` → GitHub 创建空仓库 → `git remote add origin` → `git push -u origin main`。已有团队仓库时优先 clone，不要随便 init。
